### Un projet Open Source solitaire

Nous développons un projet d'[Éditeur Markdown WYSIWYG](https://github.com/nhnent/tui.editor) depuis trois ans, avec son premier commit en mars 2015. Je suis allé travailler le 30 décembre 2017 pour publier la version 1.0 et la mettre en ligne sur [npm](https://www.npmjs.com/package/tui-editor).

À cette époque, le dépôt était aussi calme que l'air dans le bureau à la fin de l'année. Un problème posté une fois tous les quelques mois, et il semblait essayer de dire « Ce projet est open source. » Il a reçu exactement 160 étoiles GitHub en trois ans, et une pull request était difficile à espérer.

Pas de surprise. Même de la part de [Google](https://github.com/google) et [Facebook](https://github.com/facebook), il existe des projets open source qui n'ont aucun contributeur. Je suppose que certains d'entre vous ont peut-être connu la déception de voir leur précieux projet open source tomber dans le vide. De plus, nous sommes habitués à utiliser [GitHub](https://github.com/) comme notre dépôt Git gratuit. N'est-ce pas ?

Lors de la sortie de la version 1.0, je voulais que plus de personnes connaissent ce projet. Je croyais que ce projet valait plus que ce qu'il semblait à ce moment-là. J'ai fait quelques choses pour y parvenir, et en résultat, il a obtenu 2 000 étoiles en 4 jours et 3 000 étoiles en une semaine ! Maintenant, il a plus de 5 500 étoiles et des issues et pull requests pleines. Tout cela s'est passé en seulement un mois.

![Image](https://cdn-media-1.freecodecamp.org/images/fxEd5o4o2pUlorTc9inK0ShqcPRAeKbbbdIc)
_Oui. J'ai essayé d'enregistrer tout ce qui semblait important. ?_

J'ai écrit cet article pour partager mon expérience du dernier mois. Je vais également vous montrer les chiffres que j'ai enregistrés lorsque c'était possible. J'espère que cela pourra vous aider à faire en sorte que votre projet open source soit plein de contributeurs.

### Un bon fichier README attire des étoiles

Malgré les près de trois ans que nous avons passés à développer notre éditeur, personne ne semblait s'y intéresser. C'était inévitable, car les documents étaient pauvres.

Même pour une machine à laver, vous voudrez lire le manuel une fois. Vous devriez fournir encore plus de documents pour votre projet.

Le fichier README décrit l'essence du projet. La DEMO montre qu'il est prêt à être utilisé. Les exemples devraient être la première étape dans diverses exigences. Les tutoriels devraient aider quiconque à commencer.

Si vous manquez l'un de ces documents, les utilisateurs abandonneront votre projet. Pousser du code source ne fait pas de votre projet un projet open source.

Parmi les documents, le README joue le rôle le plus important. Il ne suffit pas de lister plusieurs documents. Vous, moi et la plupart des développeurs sommes paresseux. La plupart des visiteurs feront simplement défiler le README environ deux fois et partiront s'ils ne sont pas intéressés. Donc, le fichier README doit fournir la raison pour laquelle les développeurs veulent votre projet.

![Image](https://cdn-media-1.freecodecamp.org/images/VN2sBeV4QJppueU7fffqSQKqaMausO9oEKBU)
_TOAST UI Editor README.md pour les versions 0.14.0 et 1.0.0_

Voyons quelle différence la mise à jour du fichier README a apportée. Les étoiles GitHub ne sont en réalité que des signets pour les visiteurs. Mais le nombre d'étoiles GitHub vous indique à quel point les visiteurs sont intéressés par votre projet.

Les tableaux ci-dessous montrent le nombre d'étoiles et de visiteurs uniques que j'ai enregistrés avant et après la publication de TOAST UI Editor 1.0.

Il n'y avait aucune fonctionnalité ajoutée ou de rafraîchissement de design. Il n'y avait que des mises à jour des documents — principalement dans le fichier README — et un peu de refactoring. Les chiffres suivants résultent entièrement des changements dans le fichier README.

![Image](https://cdn-media-1.freecodecamp.org/images/T4viYEoUzOI5wh9T1yLa7R5jQqcvHdKwTyji)
_Nombre de visiteurs uniques et d'étoiles. Mise à jour des documents le 30/12/2017_

![Image](https://cdn-media-1.freecodecamp.org/images/GaCXGWvuhf95YHVXrVFPp14YYvmZbBAOcbSR)
_Étoile/UV est presque 5x après la mise à jour du README_

Il aurait été mieux si nous avions un enregistrement depuis le début du projet. Mais j'ai simplement enregistré les chiffres pour vérifier les tendances. Je ne peux pas faire grand-chose à ce sujet maintenant. Donc, regardons simplement les chiffres, et je laisserai cela à votre intuition.

Avant la mise à jour du fichier README, environ 4 % de tous les visiteurs donnaient une étoile. Étonnamment, après la mise à jour du fichier README, environ 20 % de tous les visiteurs donnaient une étoile. En mettant à jour le fichier README, ce projet a reçu presque **cinq fois plus d'étoiles** de la part des visiteurs !

Tout le monde dit que le fichier README est important. Mais qui savait qu'il ferait une telle énorme différence ? J'ai également été surpris de voir cela. Si vous voulez avoir des contributeurs sur un projet, vous devriez écrire un fichier README attrayant plus que tout autre chose.

Vous pouvez facilement trouver des articles sur la façon d'écrire un bon fichier README sur Google. Il est également bon d'étudier les READMEs d'autres projets listés sur [Awesome Readme](https://github.com/matiassingers/awesome-readme).

### Faites connaître votre projet dans la communauté

Notre projet avait environ dix visiteurs par jour. Si vous avez travaillé dur sur vos documents, il est temps de faire connaître votre projet.

Bien sûr, ce serait mieux si quelqu'un d'autre parlait de votre projet à votre place. Mais vous devez commencer par vous-même. Si votre projet n'est lié nulle part, il n'apparaîtra pas dans les recherches Google. Personne ne saura que votre projet existe.

#### **Parlez de votre projet à chaque occasion**

Répondez aux questions sur [StackOverflow](https://stackoverflow.com/) et [Quora](https://www.quora.com/). Parlez aux gens dans les [Groupes Facebook](https://www.facebook.com/groups/). Soumettez un lien à [HackerNews](https://news.ycombinator.com/) et [Reddit](https://www.reddit.com/). Et demandez aux [Newsletters](http://javascriptweekly.com/issues/368) de publier votre projet.

![Image](https://cdn-media-1.freecodecamp.org/images/CBGMjGs-i9eacEIZ8xKdCLs33H2rMdiez5YB)
_Dites-leur ce qu'est votre projet._

Vous pouvez également demander aux communautés de vous aider avec votre projet, ainsi que de leur parler de votre projet.

Mais ne les spammez pas aveuglément. Les communautés et les médias ont des personnalités différentes. Spammer votre projet sans comprendre chaque communauté peut entraîner la suppression de votre publication ou simplement l'ignorer. Dans le pire des cas, vous pourriez être banni de ces communautés, ou votre projet pourrait avoir une mauvaise réputation.

Soyez actif dans chaque communauté pour les comprendre. Connaître les communautés n'est pas seulement bon pour les relations publiques. Ce que vous apprenez d'elles aidera également vos projets.

Un jour, un commercial m'a dit :

> « Impliquer vos clients dans votre travail est la meilleure façon d'obtenir des clients fidèles. »

Je suis entièrement d'accord avec cela. Et je pense que nous devrions faire de même pour notre travail.

Alors que je mettais à jour le fichier README, je leur ai demandé de réviser le document. Et heureusement, deux personnes ont envoyé une pull request après avoir révisé le nouveau README. Je suis sûr que ces deux personnes sont plus intéressées par notre projet que tout autre visiteur. Je parie qu'ils ont également donné une étoile à notre projet.

### Apprendre GitHub Explore

GitHub est le meilleur endroit pour obtenir des contributeurs. Si votre projet est sur GitHub, ne cherchez pas trop loin, et apprenez-en davantage à ce sujet en premier. En haut de GitHub, vous pouvez toujours trouver un menu qui vous suit avec un mignon Octocat. L'[Explore](https://github.com/explore) est ce à quoi nous devons prêter attention. La page montre aux développeurs des projets open-source utiles et intéressants.

Explore vous donne l'opportunité de présenter votre projet à [24 millions d'utilisateurs GitHub](https://octoverse.github.com/). C'est pourquoi vous devriez maintenir votre projet open source sur GitHub.

D'autres services sont également excellents, alors ne vous méprenez pas. J'ai utilisé BitBucket pendant très longtemps pour gérer des projets personnels de manière privée. GitLab a également des fonctionnalités vraiment sympas qui, parfois, fonctionnent mieux pour moi que celles de GitHub ces dernières années. Cependant, les projets open source doivent être maintenus sur GitHub pour obtenir des visiteurs depuis Explore.

![Image](https://cdn-media-1.freecodecamp.org/images/uKJBu0akQT8hAe-QGY9CiF6HV1zP34Ud1P5O)
_GitHub Explore Trending & Topics_

Jetez un coup d'œil au type de projets qui existent, découvrez quelles sont les bonnes parties de ces projets et comment la tendance actuelle évolue. Si vous n'avez pas encore vu la page Explore, il est temps de la vérifier. Ici, vous pouvez apprendre ce que vous devez faire pour améliorer votre projet.

Lorsque vous ouvrez cette page, vous verrez les onglets [Collections](https://github.com/collections), [Topics](https://github.com/topics) et [Trending](https://github.com/trending) — chacun de ces onglets liste de beaux projets open source. Parmi eux, nous allons examiner Trending et Topics. Nous ne pouvons pas aider avec les Collections, car elles sont curatées par GitHub.

### Incitations des visiteurs depuis GitHub Trending

La section Trending de la page principale Explore montre les projets qui ont reçu le plus grand nombre d'étoiles cette semaine-là dans toutes les langues. Y figurer serait vraiment difficile.

![Image](https://cdn-media-1.freecodecamp.org/images/v6YScBoXtOlnjtj8VIqkaKSKs43RaazIX2ak)
_Trending Main 1er avec 4 761 étoiles cette semaine. Vous aurez besoin de chance pour prendre cette place._

Mais vous n'avez pas à abandonner. Réduisez la langue et la plage de dates, puis regardez les projets dans les rangs inférieurs. Il y aura un nombre réaliste que votre projet peut également atteindre.

![Image](https://cdn-media-1.freecodecamp.org/images/n4GCid-4SZ89d9Q9Mvs5T8Z5kT-El1Tp36XP)
_Trending Today JavaScript 25e avec 63 étoiles aujourd'hui. Vous pouvez le faire aussi._

Concentrez-vous sur les visiteurs dans certains jours pour figurer sur la liste en impliquant toutes les communautés dont vous faites partie. Si vous pouvez collecter suffisamment d'étoiles, votre projet peut également figurer sur la liste Trending. Le nombre d'étoiles que vous obtenez dépend de la langue de votre projet, du jour de la semaine, etc.

Si vous avez publié un nouveau projet ou ajouté des fonctionnalités au projet, c'est le moment idéal pour le mettre en avant. Ne soyez pas déçu même si votre projet est classé bas. Votre projet obtiendra toujours des visiteurs supplémentaires. Considérez la section Trending comme une incitation pour les visiteurs basée sur votre classement.

![Image](https://cdn-media-1.freecodecamp.org/images/xgjJzmJuEXJ0zEXIAAvJnpPkFePhQ2Ki2VoG)
_Visiteurs uniques le 10 janv. et le 11 janv._

Le graphique ci-dessus est une répartition des visiteurs sur GitHub Insight après que j'ai fait connaître le projet dans diverses communautés le 10 janvier. Les sites de référence dans GitHub Insight donnent des valeurs sur une période de deux semaines, vous devez donc faire quelques calculs pour obtenir les chiffres que vous voulez.

Le 10 janvier — Jour Un — presque tous les visiteurs venaient de mes propres communautés. Heureusement, une petite partie GitHub.com dans le graphique montre que notre éditeur était sur la liste Trending. Le lendemain, le 11 janvier, GitHub apportait déjà la moitié du trafic. C'est une bonne incitation à être sur la liste Trending.

![Image](https://cdn-media-1.freecodecamp.org/images/gSlAVbCTOm94GAUSM2uUt9fOGvpEuI7Ig5Yo)
_Visiteurs uniques le 12 janv._

Le lendemain, le 12 janvier, l'éditeur a été enregistré comme le n°1 Trending Today dans toutes les langues. La plupart des visiteurs venaient de GitHub. À partir de ce jour, les visiteurs provenant des publications que j'avais écrites moi-même ont considérablement diminué. D'autre part, avec l'augmentation de la visibilité, des visiteurs apparaissaient depuis des référents inattendus.

![Image](https://cdn-media-1.freecodecamp.org/images/Fkh7LnVBf-xxiw07DBLx0f0k-kvnxept2yMr)
_Visiteurs uniques le 13 janv._

Le lendemain, le 13 janvier, la plupart des visiteurs venaient de GitHub. L'éditeur a enregistré 2 000 étoiles ce jour-là, **quatre jours** après avoir été posté dans les communautés. Bien que la croissance ait diminué depuis, un mois plus tard, le [Trending for This Month](https://github.com/trending?since=monthly) apporte toujours de nouveaux visiteurs.

### Visiteurs réguliers depuis GitHub Topics

Les Topics montrent les projets dans l'ordre du nombre d'étoiles dans un topic.

Cela signifie qu'une fois que votre projet a suffisamment d'étoiles pour être listé dans un Topic, il peut toujours obtenir des visiteurs via le Topic, même si votre projet est délisté de Trending.

Il n'y a pas autant de visiteurs depuis la liste des Topics que depuis Trending. Mais cela en vaut toujours la peine.

![Image](https://cdn-media-1.freecodecamp.org/images/-pGKqUjYfdTv3sQAsvGz8S2yi4qA5UPpF-op)
_Topics du projet dans la page du dépôt_

J'ai vu de nombreux projets qui n'ont pas de Topics. Regardez l'image ci-dessus pour voir où vous pouvez configurer les Topics dans votre projet. Vous pouvez configurer n'importe quels Topics que vous voulez.

Cependant, GitHub gère une liste appelée [Featured Topics](https://github.com/topics). Par conséquent, il est préférable de choisir les topics pertinents parmi les Featured Topics pour obtenir plus de visiteurs.

![Image](https://cdn-media-1.freecodecamp.org/images/85HgGTwD9TESI-T21bArImtsKG6f4-zc4DS3)
_GitHub Explore All Featured Topics & Popular Topics_

De plus, le nombre d'étoiles nécessaires pour être listé en haut varie considérablement en fonction des topics.

Prenons le Topic JavaScript comme exemple. Il y a des projets de la taille d'un dinosaure comme FreeCodeCamp (salut les gars ?), BootStrap, React, Vue, etc. Cela rend environ 100 fois plus difficile l'ajout de votre projet à cette liste.

![Image](https://cdn-media-1.freecodecamp.org/images/9hgwX85q-YLQbcHAO3sDmEH6JOgBe7tgm4XX)
_GitHub Explore JavaScript Topic. Votre projet ne battra jamais freeCodeCamp. ?_

Notre éditeur est classé 10e dans le Topic Markdown avec 5,4k étoiles. Jetez un coup d'œil aux Topics et choisissez des topics qui ne sont pas trop compétitifs.

### La meilleure ligne unique pour la description de votre projet

![Image](https://cdn-media-1.freecodecamp.org/images/3RufjyexWtRQbUFwJE6z2pVVvHWHIXB99FDY)
_Description du projet dans la page du dépôt_

Retrouvons la description dans l'image ci-dessus. Quelle est la description du projet pour votre projet ? J'avais écrit la description de notre éditeur comme une version longue du titre (ce n'est pas une bonne idée). Avez-vous trouvé votre ligne ? Alors regardons l'image suivante.

![Image](https://cdn-media-1.freecodecamp.org/images/fN0S0mRMAk9ZnuJKr2YdiPiP4XTToB-9ILH3)
_Un projet sur la liste Trending & Topics_

Voici à quoi ressemble notre projet sur Trending et Topics. Essayez de le regarder du point de vue des visiteurs. Il y a une organisation, un nom, des photos de contributeurs, etc. La seule information utile pour décider de cliquer ou non est la description.

La description doit être la meilleure ligne unique pour décrire votre projet.

### Mot de la fin

Écrivez des documents. Soyez actif dans les communautés. Utilisez les outils de GitHub.

Les conseils que je partage dans cet article vous aideront à améliorer votre projet en tant qu'open source (en plus d'obtenir plus d'étoiles).

Mais ces conseils seuls ne peuvent pas rendre votre projet populaire sans la valeur de votre projet lui-même. J'espère que cet article vous aidera à trouver les contributeurs que votre projet mérite.

Je crois que ce projet avait déjà la valeur qu'il a aujourd'hui. Et les conseils de cet article nous ont aidés à accomplir le reste.
