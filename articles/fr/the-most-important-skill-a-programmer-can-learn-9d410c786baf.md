Par Huseyin Polat Yuruk

**Non, non, non, non, et non. Et non.**

Un grand NON. Clair comme ça.

Tout ce que vous avez à faire, c'est de rassembler ces deux lettres et de dire le mot.

Maintenant, disons-le ensemble. NONNNNNNN !

Bon début.

Mais attendez une minute. Dire NON à quoi et quand ?

Eh bien, voici le point important où la plupart des programmeurs (même les seniors) se trompent facilement.

En tant que programmeur, écrire du code fait partie intégrante de votre travail. Au cours de votre vie de programmeur, vous devrez traiter différentes demandes de code. Chaque demande vous obligera à prendre des décisions difficiles. Tout cela est CORRECT. Rien de mal à cela. C'est ce que tout le monde attend de vous, en tant que programmeur : écrire du code. Cependant, voici une question : devriez-vous écrire tout le code qui vous est demandé ?

Cette question nous amène à la compétence la plus importante qu'un programmeur peut apprendre :

> Savoir quand ne pas coder est probablement la compétence la plus importante qu'un programmeur peut apprendre. — [The Art Of Readable Code](https://www.amazon.com/Art-Readable-Code-Practical-Techniques/dp/0596802293)

Je ne pourrais pas être plus d'accord. Pourquoi ?

La programmation est l'art de résoudre un problème. Naturellement, les programmeurs sont des résolveurs de problèmes. En tant que programmeurs, lorsque nous avons un nouveau problème devant nous, prêt à être résolu, ou toute autre raison qui nous oblige à écrire des lignes de code, nous sommes excités.

Et c'est normal, car nous sommes programmeurs. Nous aimons écrire du code.

Cependant, être trop excité à l'idée d'écrire du code nous rend aveugles. Cela nous fait ignorer certains faits importants qui peuvent causer des problèmes plus grands que nous devrons résoudre à l'avenir.

Alors, quels sont ces faits importants que nous avons tendance à ignorer ?

Chaque ligne de code que vous écrivez est :

* du code qui doit être lu et compris par d'autres programmeurs
* du code qui doit être testé et débogué
* du code qui augmentera les défauts de votre logiciel
* du code qui introduira probablement de nouveaux bugs à l'avenir

Comme l'a écrit Rich Skrenta, [le code est notre ennemi](http://www.skrenta.com/2007/05/code_is_our_enemy.html) :

> Le code est mauvais. Il pourrit. Il nécessite un entretien périodique. Il contient des bugs qui doivent être trouvés. De nouvelles fonctionnalités signifient que l'ancien code doit être adapté.  
>   
> Plus vous avez de code, plus il y a d'endroits où les bugs peuvent se cacher. Les checkouts ou les compilations prennent plus de temps. Il faut plus de temps à un nouvel employé pour comprendre votre système. Si vous devez refactoriser, il y a plus de choses à déplacer.  
>   
> De plus, plus de code signifie souvent moins de flexibilité et de fonctionnalité. Cela semble contre-intuitif, mais souvent une solution simple et élégante est plus rapide et plus générale que le désordre de code produit par un programmeur de moindre talent.  
>   
> Le code est produit par des ingénieurs. Pour produire plus de code, il faut plus d'ingénieurs. Les ingénieurs ont des coûts de communication n², et tout ce code qu'ils ajoutent au système, tout en étendant ses capacités, augmente également toute une série de coûts.

C'est si vrai, n'est-ce pas ? Les programmeurs qui vous inspirent par leur productivité et leur mentalité de codage sont ceux qui savent quand dire non et quand ne pas coder. Le logiciel qui est facile à maintenir, qui dure longtemps et qui continue d'aider ses utilisateurs est celui qui ne contient aucune ligne de code inutile.

> Le meilleur code est l'absence de code, et le programmeur le plus efficace est celui qui sait quand ne pas coder.

### Comment savoir quand ne pas coder ?

Il est naturel de s'enthousiasmer lorsque vous travaillez sur un projet et que vous pensez à toutes les fonctionnalités cool que vous aimeriez implémenter. Mais les programmeurs ont tendance à surestimer le nombre de fonctionnalités dont leur projet a besoin. De nombreuses fonctionnalités restent inachevées ou inutilisées ou rendent simplement l'application trop compliquée. Vous devez savoir ce qui est essentiel pour votre projet afin d'éviter de commettre cette erreur.

> Comprendre le but de votre logiciel et sa définition centrale est la première étape pour savoir quand ne pas coder.

Permettez-moi de vous donner un exemple. Supposons que vous avez un logiciel qui n'a qu'un seul but : gérer les emails. Pour ce but, envoyer et recevoir des emails sont deux fonctionnalités essentielles pour votre projet. Vous ne pouvez pas vous attendre à ce que ce logiciel gère également votre liste de tâches, n'est-ce pas ?

Vous devez donc dire NON à toute demande de fonctionnalité qui n'est pas pertinente par rapport à cette définition. C'est le moment où vous pouvez être exactement sûr que vous savez quand ne pas écrire de code.

> Ne jamais étendre le but de votre logiciel.

Une fois que vous savez ce qui est essentiel pour votre projet, vous serez conscient la prochaine fois que vous évaluerez les demandes de code possibles. Vous connaîtrez exactement vos exigences pour écrire du code. Quelle fonctionnalité doit être implémentée ? Quel code vaut la peine d'être écrit ? Vous remettrez tout en question parce que vous saurez exactement comment le code inutile peut tuer votre projet.

> Savoir quand ne pas coder garde votre base de code petite.

![Image](https://cdn-media-1.freecodecamp.org/images/AaXgIsHTyVquQeabDnz5kMCsyEmPuMnod3E9)
_[The Art Of Readable Code](https://www.amazon.com/Art-Readable-Code-Practical-Techniques/dp/0596802293" rel="noopener" target="_blank" title=")_

Il n'y a que deux ou trois fichiers sources lorsque vous commencez votre projet. Tout semble si simple. Il ne faut que quelques secondes pour compiler et exécuter le code. Vous savez où trouver exactement ce que vous cherchez.

Ensuite, à mesure que le projet grandit, de plus en plus de fichiers sources remplissent votre répertoire. Chaque fichier de code contient des centaines de lignes de code. Pour les organiser tous, vous aurez bientôt besoin de plusieurs répertoires. Se souvenir des fonctions qui appellent d'autres fonctions est plus difficile, et le suivi des bugs nécessite un peu plus de travail. La gestion de votre projet devient difficile, et vous avez besoin de plus de programmeurs pour vous aider. Les [frais généraux de communication](https://en.wikipedia.org/w/index.php?title=Communication_overhead&action=edit&redlink=1) augmentent à mesure que le nombre de programmeurs augmente. Vous devenez de plus en plus lent.

À la fin, le projet devient énorme. Ajouter de nouvelles fonctionnalités est douloureux. Même apporter de petites modifications prend des heures. Corriger les bugs présents introduit toujours de nouveaux bugs. Vous commencez à manquer les délais.

Maintenant, la vie est une lutte pour vous. Pourquoi ?

Parce que vous ne saviez pas quand ne pas coder, vous avez dit OUI à chaque demande de fonctionnalité possible. Vous étiez aveugle. Coder quelque chose de nouveau vous a fait ignorer des faits essentiels.

C'est comme un film d'horreur, n'est-ce pas ?

C'est ce qui arrivera si vous continuez à dire OUI à tout. Sachez exactement quand ne pas coder. Éliminez tout code inutile de votre projet. Cela vous facilitera la vie et fera durer votre logiciel plus longtemps.

> L'un de mes jours les plus productifs a été de jeter 1000 lignes de code. — Ken Thompson

Je sais que savoir quand ne pas coder est si difficile. Même pour les programmeurs seniors. Peut-être que ce que j'ai écrit dans cet article est difficile à comprendre pour les programmeurs juniors, et c'est normal et compréhensible.

Je sais que vous venez de commencer votre voyage en programmation et que vous voulez écrire du code. Vous êtes si excité par cela. C'est bien. Ne perdez jamais cette excitation, mais ne ignorez jamais les faits importants. Nous les avons appris en faisant nos propres erreurs. Vous ferez des erreurs aussi, et vous apprendrez d'elles aussi. Mais au moins, vous pouvez être plus conscient si vous pouvez apprendre de notre expérience.

> Continuez à coder, mais sachez quand dire non au codage.
