Par Sergei Garcia

Tout d'abord, laissez-moi me présenter. Je m'appelle Sergei Garcia, et je suis un développeur front-end à temps plein avec 2 ans d'expérience. Durant cette période, j'ai travaillé en tant que développeur front-end pour une entreprise de conseil du Forbes 500 ainsi que pour une petite entreprise.

Cela peut ne pas sembler être beaucoup d'expérience, mais terminer ma deuxième année en tant que développeur a été un énorme jalon pour moi. Cela est dû au fait que je n'avais aucune expérience réelle en développement web — et pas beaucoup d'expérience en programmation en général au-delà de quelques formations de base en C# et Java que j'ai suivies grâce à quelques cours en ligne. Je n'avais également pas de diplôme en informatique puisque j'ai obtenu un diplôme en gestion de projets informatiques.

Je n'avais jamais écrit sur mon expérience malgré toute l'aide que j'ai reçue de ressources merveilleuses comme Medium, Stack Overflow, et les subreddits de programmation de Reddit. Alors aujourd'hui, j'ai décidé de changer cela. Aujourd'hui, je vais vous informer sur ce qui a bien fonctionné et ce qui n'a pas fonctionné, afin que si vous vous lancez dans ce voyage, vous aurez plus de chance que moi.

Je sais qu'il y a beaucoup d'articles comme celui-ci, mais peu d'entre eux discutent du processus avec le bénéfice de deux années supplémentaires de recul.

Je vais commencer par mon parcours, y compris ce qui a mal tourné en cours de route. Si vous vous intéressez uniquement à mes suggestions pour le chemin le plus court possible pour devenir développeur web à partir de zéro, n'hésitez pas à sauter à la dernière section : **Le chemin le plus court**.

Alors, sans plus tarder, commençons !

### Acquérir les bases

Après avoir décidé de me lancer dans le développement web, la première question qui m'est venue à l'esprit était « Qu'est-ce que j'apprends ? » Après avoir fait quelques recherches, j'ai fini par établir mon parcours d'apprentissage en fonction de ce que la plupart des postes de développeur web de niveau débutant demandaient, à savoir :

* JavaScript
* HTML & CSS
* Préprocesseurs CSS (Less & Sass)
* Design Réactif
* AngularJS
* Design Patterns
* Git
* NodeJS
* Task Runners

Voici comment cela s'est passé.

#### Javascript

J'ai commencé mon parcours en apprenant JavaScript grâce à [CodeSchool](https://www.codeschool.com/) (payant) et [Codecademy](https://www.codecademy.com/) (gratuit). Si vous ne connaissez pas ces plateformes, ce sont d'excellents sites qui vous permettent d'apprendre à coder en codant directement dans le navigateur.

J'ai trouvé que les ressources d'apprentissage comme celles-ci étaient les meilleures lorsque vous débutez. Sachez simplement que cette méthode d'apprentissage devient rapidement fatigante une fois que vous passez à des sujets plus avancés, car leurs algorithmes pour vérifier si vous avez correctement résolu l'exemple de code ont quelques problèmes de précision. Leurs cours d'introduction à JavaScript étaient exceptionnels et je les recommande vivement.

Une fois que j'ai maîtrisé les bases, j'ai poursuivi pour acquérir une base plus solide en JavaScript en lisant le livre [Eloquent Javascript: A Modern Introduction to Programming](http://eloquentjavascript.net/) de Haverbeke (gratuit).

Ce livre m'a été recommandé par beaucoup de gens dans les forums JavaScript comme une lecture incontournable, et pour de bonnes raisons. Ce livre était difficile — surtout si vous apprenez la programmation comme c'était mon cas à l'époque. Mais je suis content de ne pas avoir abandonné et d'avoir persévéré. Il était phénoménal grâce à l'ampleur des concepts de programmation qu'il couvre, même s'il était parfois un peu impitoyable. Quoi que vous fassiez, ne sautez pas les défis de code. Une fois que vous aurez terminé ce livre, vous pourrez enfin dire avec confiance que vous avez une bonne maîtrise de JavaScript.

Vous pouvez également apprendre jQuery de manière optionnelle (bien que je ne recommande vraiment pas de l'apprendre tout de suite — plus d'informations à ce sujet plus tard). Vous pouvez l'apprendre grâce au cours [Try jQuery](http://try.jquery.com/) de CodeSchool.

#### **HTML & CSS**

Après avoir appris JavaScript, j'ai poursuivi en apprenant les bases de HTML & CSS et du design web grâce au parcours d'apprentissage [HTML & CSS](https://www.codeschool.com/learn/html-css) de CodeSchool. Ces cours sont toujours mes préférés aujourd'hui, car le rythme est excellent, et l'ampleur générale de ce qu'ils couvrent m'a permis d'acquérir une base plus solide.

Vous pourriez également remplacer cela facilement par quelque chose comme le cours [HTML & CSS](https://www.codecademy.com/learn/web) de Codecademy et obtenir des résultats similaires. Ou si vous êtes prêt pour un défi, le cours d'Udacity [Intro to HTML and CSS](https://www.udacity.com/course/intro-to-html-and-css--ud304) est beaucoup plus complet et légèrement plus difficile.

**Bonus** : Si vous pouvez vous procurer le livre de Jon Duckett [_HTML and CSS: Design and Build Websites_](https://www.amazon.com/HTML-CSS-Design-Build-Websites/dp/1118008189/ref=sr_1_3?ie=UTF8&qid=1471443779&sr=8-3&keywords=john+ducket), c'est aussi un excellent point de départ pour apprendre HTML & CSS (avec une touche de design web). Il est très bien noté (4,7/5 sur Amazon), offre une solide introduction au monde du développement web. C'est un beau livre grâce à son design épuré avec de grandes lettres et des pages colorées. Je reviens souvent pour l'admirer.

#### Less / Sass

Pour ceux qui ne connaissent pas, Less & Sass sont des transpileurs CSS qui vous permettent d'écrire du CSS de manière plus élégante. Cela vous permet de faire des choses qui ne sont normalement pas supportées, comme l'imbrication des règles CSS. Une fois terminé, ces transpileurs CSS « compilent » votre code et le convertissent en CSS normal.

Il existe 2 principaux transpileurs CSS actuellement : **Less** et **Sass**. Sass est le plus populaire, mais j'ai trouvé que l'apprentissage de Less en premier était plus facile, principalement parce que l'utilisation de Sass sur votre ordinateur nécessite également l'installation de Ruby, ce que je n'appréciais pas.

Vous pouvez obtenir un aperçu rapide, mais complet de Less en utilisant le [compilateur Less en ligne de WinLess](http://winless.org/online-less-compiler) et ses exemples de code pour voir comment votre code Less se transformerait en CSS. Vous pouvez également essayer Sass en ligne en utilisant [SassMeister](http://www.sassmeister.com/) (bien que cela n'inclue pas d'exemples de code).

Peu importe que vous appreniez Less ou Sass en premier. Ils sont extrêmement similaires, donc une fois que vous connaissez l'un, vous connaissez pratiquement l'autre. Vous pouvez trouver une excellente comparaison rapide entre Less et Sass dans l'article de Shelby Moulden [Comparison between LESS & SASS](http://shelbymoulden.com/blog/?post=comparison-between-less-and-sass).

#### Responsive Design

J'ai initialement appris le responsive design et Bootstrap en utilisant le parcours HTML & CSS de Codeschool, mais j'ai récemment trouvé le cours d'Udacity par Google sur les [Responsive Web Design Fundamentals](https://www.udacity.com/course/responsive-web-design-fundamentals--ud893) fantastique pour couvrir les bases et au-delà de manière beaucoup plus complète que Codeschool.

Vous pouvez faire du responsive design sans aucun framework supplémentaire, mais c'est beaucoup plus facile avec l'aide d'un framework responsive comme Bootstrap. La [documentation officielle](http://getbootstrap.com/css/) de Bootstrap est très bien faite, donc vous ne devriez avoir aucun problème pour commencer avec.

Si vous avez du mal à comprendre ses principes de base, lisez l'article de blog de Froont sur les [9 principes de base du responsive web design](http://blog.froont.com/9-basic-principles-of-responsive-web-design/). Il contient de belles animations claires et simples qui aident à illustrer visuellement les principes du responsive web design.

#### AngularJS

Je ne savais pas vraiment ce qu'était exactement AngularJS à l'époque, mais je savais que tout le monde en parlait, et que si je voulais devenir développeur web, je devais l'apprendre. J'ai trouvé que la vidéo de Google Developers [Design Decisions in AngularJS](https://www.youtube.com/watch?v=HCR7i5F5L8c) fournissait la meilleure vue d'ensemble générale de ce qu'était AngularJS et de la manière dont il améliorait la création d'applications web.

J'ai d'abord pensé à apprendre AngularJS grâce à leur documentation officielle, mais cela s'est avéré être une mauvaise idée. La documentation n'était pas très accessible aux débutants, et la mise en page encombrée la rendait difficile à lire et à comprendre.

J'ai ensuite poursuivi l'apprentissage d'AngularJS grâce à Codeschool. Avec mon expérience positive sur les cours JavaScript et CSS également de là, je m'attendais à rien de moins qu'un excellent cours. Je me trompais. Le cours était un désastre dès le départ, car l'algorithme utilisé pour vérifier si vous aviez bien compris l'exemple de code ne fonctionnait parfois pas correctement et marquait votre solution clairement correcte comme incorrecte. Il y avait même des fois où il suffisait de rafraîchir la page pour corriger le système de validation défectueux. Quant au contenu du cours, il n'était pas non plus génial. Il faisait un travail correct pour expliquer les composants de base d'une application AngularJS, mais il faisait un travail terrible pour intégrer ceux-ci dans une application réelle, me laissant avec beaucoup plus de questions que je n'en avais au départ.

Après quelques recherches sur les forums, je suis tombé sur [Egghead.io](https://egghead.io/) (gratuit / payant) où j'ai eu beaucoup plus de chance. Leur matériel de cours était beaucoup plus clair, plus concis et plus complet, ce qui a permis une bien meilleure expérience. Sans parler du fait qu'en plus de leurs cours, ils ont des leçons de 2 à 5 minutes qui couvrent des sujets importants. (Par exemple : Qu'est-ce qu'un contrôleur ? Qu'est-ce qu'un filtre ? Qu'est-ce que $scope ?) Cela rend vraiment facile la compréhension des bases. Ils ont également quelques vidéos qui nécessitent un paiement, mais elles couvrent généralement des sujets plus avancés sur Angular que vous n'aurez pas besoin d'apprendre avant plus tard. J'ai suivi leur cours [AngularJS Fundamentals](https://egghead.io/courses/angularjs-app-from-scratch-getting-started) et j'ai été entièrement satisfait des résultats (et je suis également devenu un grand fan des cours d'Egghead.io dans le processus).

#### Design Patterns

Les design patterns sont essentiellement des solutions de code réutilisables qui peuvent être utilisées à plusieurs reprises pour résoudre des problèmes logiciels courants. Avoir une base solide sur ce sujet vous rendra beaucoup plus compétitif en tant que développeur logiciel dans n'importe quel langage de programmation. Cela vous permettra également de mieux comprendre le code des autres, car vous identifierez rapidement quel design pattern ils ont utilisé dans leur code pour mieux le comprendre.

J'ai trouvé que les 2 meilleures sources pour apprendre cela sont les [JavaScript Design Patterns](http://www.dofactory.com/javascript/design-patterns) de doFactory et le livre [Learning JavaScript Design Patterns](https://addyosmani.com/resources/essentialjsdesignpatterns/book/) d'Addy Osmani. J'ai trouvé doFactory beaucoup plus facile à comprendre, tandis que le livre d'Addy Osmani était beaucoup plus complet.

#### Chrome DevTools

Chrome est l'un des outils les plus puissants pour un développeur web. Plus vous le maîtrisez tôt, plus vous pouvez gagner de temps par la suite. Le cours gratuit de Codeschool [Explore and Master Chrome DevTools](http://discover-devtools.codeschool.com/) fait un excellent travail pour les introduire.

#### Git (Contrôle de version)

Ah, Git — l'outil dont je ne savais pas que j'avais besoin jusqu'à ce que je découvre ce qu'il pouvait faire. Git vous permet essentiellement de suivre les modifications que vous apportez à votre code afin que, si les choses tournent mal, vous puissiez revenir à un point précédent dans le temps. Il vous permet également de voir l'historique de votre code.

J'ai trouvé que le cours gratuit de CodeSchool [Try Github](https://try.github.io/levels/1/challenges/1) était une manière conviviale de commencer. La [formation Git d'Atlassian](https://www.atlassian.com/git/tutorials/) était excellente pour couvrir les commandes plus avancées disponibles. Le [parcours d'apprentissage Git](https://www.codeschool.com/learn/git) de Codeschool est également excellent pour couvrir les fondamentaux de Git.

#### NodeJS

Il n'a pas fallu longtemps avant que j'apprenne qu'avoir une compréhension de base de NodeJS m'aiderait grandement dans ma quête de devenir développeur web (plus d'informations à ce sujet bientôt).

J'ai essayé les cours de Codeschool sur Node, mais je les ai trouvés vraiment pauvres en contenu. J'ai trouvé [NodeSchool.io](http://nodeschool.io/) être un bien meilleur enseignant pour acquérir les bases, et c'était amusant ! J'ai adoré l'approche pratique qu'il offrait, similaire à Codeschool et Codecademy — avec l'amélioration supplémentaire que j'exécutais vraiment NodeJS.

#### Task Runners (Grunt & Gulp)

Grunt et Gulp ont été une grande surprise pour moi dans le sens où je ne savais pas que des outils comme ceux-ci existaient — mais je suis extrêmement heureux qu'ils existent ! Essentiellement, ces task runners vous permettent d'automatiser des tâches courantes. Par exemple, vous vous souvenez de Less/Sass ? Normalement, vous devriez exécuter manuellement le compilateur CSS chaque fois que vous faites une modification pour qu'il compile le CSS, puis mettre à jour le navigateur. En utilisant un task runner, vous pouvez le configurer pour surveiller vos fichiers Less/Sass pour les modifications, et lorsqu'il détecte un changement, compiler votre CSS et rafraîchir automatiquement le navigateur. Cela est extrêmement utile pour réduire votre temps de développement.

Il existe 2 principaux task runners actuellement : Grunt et Gulp. Bien qu'ils fassent exactement la même chose, ils fonctionnent de manière très différente, Grunt étant beaucoup plus verbeux et orienté configuration, et Gulp étant plus court à écrire et préférant le code à la configuration.

Connaître NodeJS vous aidera à écrire de meilleurs fichiers Grunt et Gulp puisque **les deux fonctionnent sur NodeJS**. Vous pouvez choisir celui que vous voulez, mais j'ai trouvé Gulp beaucoup plus facile à apprendre et à écrire. Je le préfère encore aujourd'hui en raison de son approche minimaliste — mais puissante — basée sur les pipes.

J'ai trouvé que les cours de Scotch.io sur [Grunt](https://scotch.io/tutorials/a-simple-guide-to-getting-started-with-grunt) et [Gulp](https://scotch.io/tutorials/automate-your-tasks-easily-with-gulp-js) étaient parmi les meilleurs disponibles.

### Les défis auxquels j'ai été confronté lors de mon premier emploi

Une fois que j'ai couvert les bases du développement web, j'étais prêt pour mon premier entretien de développement web pour un poste de niveau débutant. Je ne vais pas entrer dans les détails de l'entretien puisque ce n'est pas le sujet principal de cet article. Mais je dirai que l'on m'a dit que mes connaissances relativement solides en JavaScript m'ont aidé à obtenir le poste. (Merci, Eloquent JavaScript !)

Je dois dire que j'étais assez nerveux pour mon premier projet. Il s'agissait de créer des composants web réutilisables avec HTML, CSS et JavaScript, ainsi que Bootstrap, Sass, Grunt comme outils.

Les deux plus grandes erreurs que j'ai commises au début étaient :

1. **La peur de l'échec.** Parce que j'étais le nouveau, j'avais constamment peur que mon code soit incorrect ou mal fait, donc je passais beaucoup de temps à tout vérifier et à adhérer aux meilleures pratiques de codage. À cause de cela, je tentais rarement des solutions de nouvelles manières créatives par peur que cela ne fonctionne pas correctement à la fin. Cela a effectivement stoppé ma motivation à apprendre de nouvelles choses.
2. **Faire des choses parce que la personne « X » qui en sait plus que moi l'a dit.** J'ai fait cela beaucoup au début. Bien que ce ne soit pas complètement faux, faire des choses d'une certaine manière seulement parce qu'un expert « X » sur le sujet l'a dit — sans savoir pourquoi — m'a conduit à ne pas vraiment savoir quand et pourquoi les choses étaient faites de cette manière. J'ai rapidement appris qu'il y avait des exceptions à tout, et que **vous devez toujours connaître la raison derrière les meilleures pratiques.**

Heureusement, j'ai eu un chef d'équipe compréhensif lors de mon premier projet qui m'a aidé à surmonter ces problèmes. Il m'a constamment motivé à essayer de nouvelles choses, même si parfois les choses tournent mal. Il m'a également dit de tout remettre en question — même ses enseignements.

Avec le temps, j'ai appris ma leçon. Depuis lors, j'ai toujours été une personne qui se réjouit d'essayer de nouvelles choses. Je cherche toujours à comprendre pourquoi les meilleures pratiques existent, quand elles sont appropriées et quand elles ne s'appliquent pas à une situation.

L'utilisation d'AngularJS dans un projet réel a également posé un défi de taille pour moi. Cela était principalement dû au fait que beaucoup de choses que je faisais avec, je les faisais sans comprendre pleinement pourquoi elles se produisaient. Je les considérais comme de la « magie Angular ».

Il y a eu de nombreuses fois où j'aurais aimé savoir comment Angular fonctionnait réellement, mais la documentation était effrayante.

J'ai finalement découvert un livre incroyable intitulé [Build Your Own AngularJS](http://teropa.info/build-your-own-angular/). Je ne l'ai pas lu en entier, mais la lecture de la section sur les Scopes et les Watchers et leur fonctionnement a vraiment révélé que la magie derrière Angular n'était pas vraiment de la magie. C'était simplement une manière ingénieuse de maintenir la liaison de données en utilisant la vérification de cohérence et les portées imbriquées. Je recommande vivement ce livre à toute personne cherchant à comprendre pleinement AngularJS.

L'autre défi auquel j'ai été confronté un an plus tard était la rapidité avec laquelle le développement web progressait. Je venais de maîtriser AngularJS et Grunt, et je me sentais fier et puissant — pour bientôt découvrir que Gulp et ReactJS étaient à l'horizon. Et un an plus tard, après les avoir appris, Webpack a commencé à gagner du terrain, et j'ai dû l'apprendre aussi. Comme vous pouvez l'imaginer, une grande partie de moi était assez déçue de la rapidité avec laquelle certaines de mes connaissances sont devenues obsolètes. Mais un collègue m'a bientôt éclairé en me disant quelque chose qui a changé ma vision des bibliothèques et des frameworks pour toujours :

> **« Les bibliothèques et les frameworks peuvent devenir obsolètes, mais les concepts et les solutions qu'ils proposent survivent souvent à l'épreuve du temps. »**

Il avait raison. AngularJS peut être devenu obsolète, mais comprendre pleinement la magie derrière celui-ci m'a aidé à mieux comprendre l'architecture des composants web de React, qui a amélioré le concept des Directives d'Angular. Cela m'a également aidé à comprendre comment ReactJS a gagné autant de popularité, ainsi que le type d'avenir qui l'attendait.

Je ne me souviens pas avoir été confronté à d'autres défis majeurs dans mes projets suivants. Mais ce que je dirai, c'est que sur les 2 ans où j'ai fait du développement web, la chose numéro 1 qui m'a aidé à réussir (selon mes propres collègues) était mon excitation et ma puissante motivation à toujours être à l'affût de nouvelles choses à apprendre. J'ai rapidement découvert que c'était une combinaison gagnante avec le développement web puisque les choses changent vraiment, vraiment vite ici, avec de nouveaux frameworks et bibliothèques émergent constamment.

De l'autre côté de la médaille, l'autre chose qui m'a énormément aidé — et quelque chose que j'ai découvert assez récemment en fait — était de comprendre ce qu'il **ne fallait pas** apprendre. Cela est devenu crucial dans mon processus de devenir un meilleur développeur web.

Il n'est pas rare de voir des gens critiquer le rythme anormalement rapide de l'évolution des technologies web, ou comment une nouvelle bibliothèque ou un nouveau framework JavaScript sort presque tous les jours. Mais avec le temps, j'ai vu la lumière et j'ai enfin compris :

> _Vous n'avez pas à apprendre chaque nouvelle bibliothèque ou framework qui sort._

Souvent, c'est une excellente idée de faire une simple application exemple « hello world » pour voir ce qu'un framework offre. Ensuite, vous pouvez passer à autre chose. Mais généralement, vous devriez essayer de vous concentrer sur ce qui convient le mieux aux besoins de votre projet. Cela peut être difficile au début, mais heureusement, des endroits comme Stack Overflow, Medium et Reddit existent où vous pouvez trouver des discussions utiles entre frameworks et déterminer lesquels correspondent le mieux à vos cas d'utilisation spécifiques.

### Aller plus loin

Au cours des années à venir, j'ai continué à m'améliorer de manière continue dans les domaines suivants.

#### JavaScript

Une fois que vous avez terminé Eloquent JavaScript, il est plutôt facile de dire et de sentir que vous avez maîtrisé JavaScript, mais ensuite vient [You Don't Know JS](https://github.com/getify/You-Dont-Know-JS) et il vous détruit absolument (ou du moins, c'est ce qu'il m'a fait). Cette série de livres (gratuits, soit dit en passant) m'a été mentionnée plusieurs fois par quelques développeurs web seniors au bureau comme _le livre_ à lire, et que seulement après l'avoir lu, je pourrais dire que je connais pleinement JavaScript. Ils avaient raison, car page après page, il a continué à me surprendre quant à la complexité réelle de JavaScript, ainsi que de nombreux pièges courants que les personnes inexpérimentées et expérimentées sans une compréhension appropriée de JavaScript pourraient avoir.

Lire cette série de livres a vraiment ouvert mon esprit, et je la recommande également à toute personne souhaitant se considérer comme un développeur JavaScript expert. Une fois que vous avez terminé cela, il y a 2 ressources supplémentaires que je recommande vivement pour obtenir une connaissance encore plus avancée de JavaScript ;

* [JavaScript, The Better Parts](https://vimeo.com/97419177) : Une conférence incroyable de D. Crockford qui parle des plus grandes faiblesses de JavaScript, de ses « Foot Guns », et de la manière de les utiliser comme ses forces.
* [The Two Pillars of JavaScript](https://medium.com/javascript-scene/the-two-pillars-of-javascript-ee6f3281e7f3#.91eor7lrd) : un article solide de l'écrivain JavaScript reconnu sur Medium, Eric Elliott, qui parle des deux grands piliers de JavaScript : l'héritage prototypal et la programmation fonctionnelle.

Une fois que vous avez une compréhension profonde de JavaScript, passez à ECMAScript 2015 (également connu sous le nom d'ES6), la dernière norme JavaScript actuelle. L'article de Smashing Magazine [ECMAScript 6 (ES6): What's New In The Next Version Of JavaScript](https://www.smashingmagazine.com/2015/10/es6-whats-new-next-version-javascript/) est une excellente revue courte de ce qui est nouveau dans ES6. Vous pouvez essayer ES6 dans le navigateur en utilisant le [transpileur en ligne de Babel](https://babeljs.io/repl/).

#### CSS

CSS peut devenir désordonné et désorganisé très, très rapidement. Il y a eu plusieurs méthodologies différentes proposées pour écrire du CSS plus propre, mais deux se distinguent que je vous recommande vivement de lire dès que possible pour rester compétitif :

* [SMACSS](https://smacss.com/) : Architecture CSS scalable et modulaire. Un guide flexible pour développer des sites petits et grands.
* [BEM](http://getbem.com/) : une méthodologie qui vous aide à obtenir des composants réutilisables et le partage de code en front-end.

Je préfère personnellement SMACSS pour son apparence plus propre, mais certaines entreprises et frameworks CSS utilisent encore BEM, donc cela vaut la peine de connaître les deux.

Vous devriez également commencer à vous concentrer sur les performances de votre CSS. L'article de Smashing Magazine [Managing Mobile Performance Optimization](https://www.smashingmagazine.com/2016/03/managing-mobile-performance-optimization/) et l'article de HTML5 Rocks [High Performance Animation](http://www.html5rocks.com/en/tutorials/speed/high-performance-animations/) ont fait un travail solide pour fournir un bon départ sur ce sujet. Une lecture rapide de ces deux articles devrait vous donner une base solide.

#### JavaScript Bundlers

À ce stade, vous devriez avoir une solide compréhension de Grunt ou Gulp. L'étape suivante consiste à ajouter un bundler JavaScript à votre task runner, ce qui permettra une organisation plus modulaire de votre application JavaScript.

Les deux principaux acteurs actuellement sont :

* [Browserify](http://browserify.org/) : vous permet de requérir des modules dans le navigateur en regroupant toutes vos dépendances.
* [Webpack](https://webpack.github.io/) : essentiellement Browserify sur stéroïdes. Plus difficile à configurer et à mettre en place.

Le mini-cours de Scotch.io [Getting Started with Browserify](https://scotch.io/tutorials/getting-started-with-browserify) peut vous donner un bon départ avec Browserify, tandis que l'article de David Fox Powell [Why Can't Anyone Write a Simple Webpack Tutorial](http://Why%20Cant%20Anyone%20Write%20a%20Simple%20Webpack%20Tutorial?) est une excellente introduction amusante à lire sur webpack.

Personnellement, je n'ai pas passé beaucoup de temps à utiliser webpack, mais pendant le temps que j'ai passé avec, je dois dire que c'était incroyable — même si c'est légèrement plus difficile à configurer. Si vous débutez, je vous recommande Browserify car il est beaucoup plus simple à configurer. Sachez simplement que webpack est l'avenir et ce que les grands projets commencent à utiliser.

#### ReactJS

ReactJS gagne rapidement en popularité et ne semble pas ralentir — au point que les gens se demandent « React tue-t-il Angular ? »

Le guide de Scotch.io [Learning React.js: Getting Started and Concepts](https://scotch.io/tutorials/learning-react-getting-started-and-concepts) offre une vue d'ensemble solide de React. Une fois que vous avez terminé cela, continuez avec le cours d'Egghead.io sur [React Fundamentals](https://egghead.io/courses/react-fundamentals) où vous construirez une application ReactJS entièrement fonctionnelle puis la migrerez vers la syntaxe ES6. Vous pouvez suivre avec la [documentation officielle de ReactJS](https://facebook.github.io/react/docs/getting-started.html) qui est très bien faite et vous permettra de la maîtriser pleinement.

Puisque React n'est que la vue, il est fortement recommandé d'apprendre Redux. La plupart des cours sur Redux sont un peu complexes à mon avis, mais [CSS Tricks Leveling Up with React: Redux](https://css-tricks.com/learning-react-redux/) offre un bon équilibre entre simplicité et information pour commencer avec Redux.

Vous avez peut-être également entendu parler de Flux à ce stade, mais si vous vous demandez pourquoi vous devriez utiliser Redux plutôt que Flux, consultez la question sur Stack Overflow [Why use Redux over Facebook Flux?](http://stackoverflow.com/questions/32461229/why-use-redux-over-facebook-flux) qui a été répondue par le créateur de Redux !

### Réflexion sur mes erreurs et ce que j'ai appris

J'ai fait beaucoup d'erreurs au cours de mes 2 ans d'apprentissage du développement web. Dans l'ensemble, je pense que ma plus grande erreur a été de ne pas maîtriser les bases avant de passer aux bibliothèques et frameworks. Je suppose que cela s'applique à presque tous les langages de programmation, mais à mon avis, cela s'applique encore plus à JavaScript. Cela est dû au fait que, de nombreuses manières, JavaScript est un langage cassé et contient beaucoup de « Foot Guns » (vous devriez en avoir entendu parler si vous avez regardé la conférence de D. Crockford sur « JavaScript, the better parts » que j'ai mentionnée plus tôt). Ceux-ci peuvent rendre la vie insupportablement difficile si vous ne les comprenez pas pleinement.

Je me souviens une fois avoir été bloqué sur un problème AngularJS avec $scope qui m'a pris 3 jours à déboguer, pour découvrir qu'il ne s'agissait même pas d'un problème AngularJS, mais d'un problème JavaScript que j'avais causé moi-même en ne comprenant pas comment _this_ fonctionne.

#### Code propre

Il est étrange que je ne voie pas cela mentionné aussi souvent. Je ne me souciais pas toujours d'écrire du code propre, mais honnêtement, c'est l'une des choses dont je suis le plus fier d'avoir apprises. Cela est dû au fait que tout le monde aime se plaindre de la manière dont leur dernier emploi avait l'une des pires, des bases de code les plus laides au monde. Alors pourquoi personne ne peut-il parler de la manière dont leur dernier emploi était génial ? Comment leur code est devenu si propre et bien fait qu'ils en étaient fiers ?

C'est une tendance que j'aimerais changer, et je crois qu'une différence peut être faite si suffisamment de personnes poussent pour cela. Efforcez-vous de rendre les noms de variables et de fonctions compréhensibles en anglais, même si vous devez écrire un peu plus. Ne pas le faire ne conduira qu'à devoir le documenter manuellement à un moment donné dans le futur pour le rendre plus clair. Cela rendra également votre base de code globale plus difficile à comprendre pour les nouveaux développeurs et vous-même. Oui, vous-même. Pourquoi vous-même ? Parce que si vous n'appliquez pas le code propre, qu'est-ce qui vous fait penser que vos collègues devraient l'appliquer et écrire du code propre pour que vous puissiez facilement le comprendre ? Dirigeons par l'exemple.

Et si ce n'est pas une incitation suffisante, les gens reconnaissent et valorisent souvent les rédacteurs de code propre. Vous constaterez qu'en écrivant du code propre, vos collègues et amis apprécieront encore plus de travailler avec vous, et à leur tour, vous vivrez une vie plus heureuse.

#### jQuery

Certains d'entre vous pourraient remarquer que je n'ai pas non plus mis beaucoup d'emphase sur jQuery. Cela est dû au fait que, selon mon expérience, j'ai trouvé que jQuery m'a fait plus de mal que de bien au début. Certains d'entre vous pourraient ne pas être d'accord, mais veuillez me laisser expliquer : Lorsque je l'ai appris pour la première fois, l'idée générale que j'ai comprise était que jQuery était partout et que vous pouviez l'utiliser pour presque tout. À cause de cela, je me suis habitué à utiliser jQuery pour presque tout, et pour tout problème que je rencontrais, je cherchais une solution qui utilisait jQuery.

Ne vous méprenez pas, jQuery était génial pendant le temps où je l'ai utilisé, si génial en fait, que j'ai ignoré aveuglément que 90 % de ce que je faisais avec jQuery pouvait être fait nativement dans les navigateurs modernes avec une syntaxe tout aussi facile.

Vous pourriez maintenant penser : « Alors, qu'est-ce qui ne va pas avec cela ? jQuery ne pèse pas si lourd de toute façon et en l'utilisant, vous finissez par écrire moins de code que si vous faisiez les choses nativement. » Mais utiliser jQuery au lieu des API natives n'était pas le problème. Le problème était que toute ma façon de penser et toutes les solutions aux problèmes courants que je connaissais jusqu'à ce point nécessitaient jQuery pour fonctionner. Et cela est devenu un énorme problème lorsque j'ai obtenu mon premier projet et qu'on m'a dit que jQuery n'était pas une dépendance.

Utiliser jQuery m'a rendu inutile sans lui et m'a fait ignorer complètement les méthodes et solutions natives qui ont toujours existé. Cela a également rendu toutes mes solutions moins portables, car leur utilisation nécessitait jQuery.

Depuis lors, je me suis efforcé de ne pas utiliser jQuery sauf si cela est absolument nécessaire et apporte une réelle amélioration en termes d'efficacité et de lisibilité à notre base de code (par exemple, une manipulation intensive du DOM).

Encore une fois, ne vous méprenez pas, jQuery est génial, mais si je pouvais remonter le temps et rencontrer mon ancien moi qui apprenait le développement web, je me conseillerais fortement de ne pas apprendre jQuery du tout jusqu'à ce que j'aie appris à faire les choses sans lui. Si vous avez du mal à faire la transition comme je l'ai fait, consultez [You Might Not Need jQuery](http://youmightnotneedjquery.com/).

#### Cours

En ce qui concerne le matériel de cours ; bien que beaucoup de cours de CodeSchool étaient exceptionnels (la branche HTML & CSS était particulièrement fantastique), même si quelques-uns de leurs cours sur les frameworks étaient un peu décevants (AngularJS, BackboneJS, etc.).

J'ai également suivi beaucoup de cours sur [Pluralsight](https://www.pluralsight.com/), que je n'ai pas mentionnés car après tout ce temps, j'en suis arrivé à la conclusion que choisir leur parcours d'apprentissage est globalement **une mauvaise idée et peu fiable**. Puisque leurs cours sont créés par des enseignants qui ne sont pas toujours (à mon avis) très bons en enseignement, j'ai trouvé que la qualité de leurs cours fluctuait énormément puisque leurs normes de qualité de cours sont inexistantes. J'ai eu des cours où même la personne donnant le cours semblait s'endormir. Et honnêtement, je n'ai pas l'attention nécessaire pour continuer à prêter attention à un cours de 6 à 10 heures — et beaucoup d'entre eux durent aussi longtemps, sinon plus.

J'ai passé un bon 80 à 100 heures de formation sur Pluralsight, et honnêtement, j'en veux une bonne partie en retour. Ne vous méprenez pas, j'ai eu quelques cours incroyables sur Pluralsight, mais leur focus sur la quantité plutôt que sur la qualité m'a vraiment fait perdre mon temps. J'aurais pu apprendre tellement plus si j'avais suivi des cours de meilleures sources comme Egghead.io et CodeSchool, où ils valorisent plus la qualité que la quantité.

La seule raison pour laquelle je pourrais penser que quelqu'un utilise Pluralsight est de suivre un cours que aucun autre site n'a dans une technologie plus obscure (comme Installshield, ou Xamarin), ou de suivre quelques cours très spécifiques qui sont reconnus comme étant très bien reçus et évalués (par exemple, les fondamentaux d'Angular de John Papa).

Dans l'ensemble, si vous souhaitez utiliser Pluralsight, assurez-vous de suivre des cours sélectionnés à la main par quelqu'un qui les a suivis en premier et qui sont reconnus comme étant de haute qualité et utiles.

J'ai également récemment essayé la formation [Team Treehouse](https://teamtreehouse.com) et je dois dire que je suis impressionné par la qualité de leurs cours, rivalisant même avec celle de CodeSchool, et leur matériel de cours est très complet.

Après avoir parcouru les parcours d'apprentissage HTML, CSS et JavaScript là-bas, je vois que vous pourriez facilement acquérir les bases de presque tout. Vous ne me croyez pas ? Jetez un coup d'œil à [leurs parcours d'apprentissage](https://teamtreehouse.com/tracks) et dites-moi que ce n'est pas incroyable. Oui, c'est un peu cher à 30 USD par mois, mais à mon avis, cela en vaut vraiment la peine. (Je paie pour cela maintenant pour apprendre WordPress puisque j'en ai besoin pour un projet freelance et le matériel est excellent).

#### Un mot sur les cours payants

J'ai ressenti le besoin de parler de cela puisque j'ai remarqué le consensus général selon lequel vous pouvez apprendre la programmation sans payer un centime et être aussi compétitif que quelqu'un qui a payé pour un cours. Bien que cela soit vrai, je ne peux pas insister assez sur la valeur de payer pour le bon cours. Bien sûr, une grande partie du matériel de cours le plus précieux dont j'ai parlé est gratuit, mais une grande partie est également payante. Principalement parce que parfois, vous ne pouvez tout simplement pas battre le fait d'avoir quelqu'un qui vous explique soigneusement les choses de manière visuelle.

Oui, il existe des cours payants d'apprentissage terribles que je déconseillerais car leur proposition de valeur est questionable (voir Pluralsight), mais d'autres comme Egghead.io, CodeSchool et Team Treehouse offrent un excellent rapport qualité-prix, malgré leur abonnement mensuel relativement coûteux (25-30 $ par mois). De plus, ils offrent tous des essais gratuits de 7 à 15 jours afin que vous puissiez voir lequel fonctionne le mieux pour vous.

Si vous jouez bien vos cartes, payer 1 à 2 mois de l'un d'eux peut facilement vous apporter des connaissances que vous n'auriez autrement obtenues qu'après être tombé sur d'innombrables articles et billets de blog sur une année. Ils sont honnêtement si bons.

Donc oui, ils ne sont pas nécessaires, mais si vous pouvez vous permettre au moins un mois, vous pouvez être sûr que cela vous donnera un avantage certain.

#### La recette secrète du succès

J'ai rencontré beaucoup de développeurs au cours des 2 dernières années où j'ai été développeur web. Au cours de mon parcours, j'ai rencontré quelques développeurs qui se distinguaient vraiment — des développeurs qui étaient clairement dans une ligue à part et auxquels moi et tout le monde admirait. J'ai trouvé que ces individus partageaient plusieurs caractéristiques, que je souhaite partager avec vous maintenant. Ce sont, à mon avis, les ingrédients secrets pour être un développeur web réussi :

* **Aimez ce que vous faites.** C'est simplement la caractéristique la plus importante de toutes. Si vous n'aimez pas ce que vous faites (que ce soit le style CSS ou JavaScript), cela se verra vraiment dans ce que vous faites. Ceux qui sont passionnés par ce qu'ils font se distinguent souvent clairement de la foule.
* **Soyez généreux et partagez vos connaissances.** Il est très facile de vouloir garder secret ce nouveau hack CSS/JavaScript que vous avez trouvé et qui résout les problèmes du projet, mais s'il vous plaît, ne le faites pas. Les personnes qui partagent le plus leurs connaissances sont souvent les plus précieuses, car elles peuvent être placées dans n'importe quelle équipe et améliorer sa qualité de manière significative.
* **Soyez toujours à l'affût des nouvelles choses.** La plupart des développeurs réussis que j'ai rencontrés partagent ce trait commun. Que ce soit en lisant des blogs, en passant beaucoup de temps dans des discussions liées à la programmation, ou même en parlant de ce qui est nouveau dans le développement web pendant les pauses déjeuner. Être à l'affût des nouvelles choses tout le temps permet aux meilleurs développeurs de toujours rester en avance sur la courbe.

#### Bonus : Mes outils

Je pensais qu'il serait également agréable de partager certains des outils que j'ai découverts (certains bien connus, d'autres moins) qui ont facilité ma vie en tant que développeur web, les voici.

* [Jetbrains Webstorm](https://www.jetbrains.com/webstorm/) : IDE de développement web complet. (Mon éditeur de choix) Payant, mais offre une licence gratuite d'un an pour les étudiants.
* [Atom.io](https://atom.io/) : Éditeur de texte hautement extensible avec des fonctionnalités similaires à un IDE rivalisant avec Webstorm. Gratuit.
* [Sublime Text](https://www.sublimetext.com/) : Éditeur de texte ultra-rapide avec support de plugins et une apparence esthétiquement plaisante. * [Cloud 9](https://c9.io/) : Environnement de développement et IDE basé sur le cloud avec support Git qui fonctionne sur Linux. Idéal pour programmer à distance et tester NodeJS ou d'autres choses côté serveur sans avoir besoin d'installer quoi que ce soit sur votre machine.

* [Vanilla List](http://www.vanillalist.com/) : Un dépôt de plugins et bibliothèques JavaScript utilisant uniquement JavaScript vanilla (ce qui signifie qu'ils ne nécessitent aucune bibliothèque pour fonctionner, comme jQuery).

* [PublicAPIs](https://www.publicapis.com/) : Vous vous êtes déjà demandé quelles API publiques existent ? Ne cherchez pas plus loin !
* [Gravit.io](https://www.gravit.io) : Application de design basée sur le cloud rivalisant avec Adobe Illustrator. (Gratuit !) Utile pour les maquettes rapides et le design web.
* [Adobe Kuler](https://color.adobe.com) : Application web pour vous aider à créer des combinaisons de couleurs harmonieuses pour n'importe quel site web. Dispose également d'une vitrine « Explorer » de palettes de couleurs créées par d'autres designers ainsi qu'un système de classement pour vous inspirer.
* [Name that color](http://chir.ag/projects/name-that-color) : Arrêtez de passer beaucoup de temps à essayer de nommer vos variables de couleur dans less/sass et utilisez simplement leur nom légitime avec cette application web.
