Par Fagner Brack

#### Taper du code peut vous aider à développer un état d'esprit d'apprentissage

Vous voulez casser du code ? Changez-le rapidement avant même de comprendre ce qu'il fait.

Maintenant, vous pratiquez la programmation Cargo Cult — un style de développement logiciel où vous ignorez comment un morceau de code fonctionne et sa relation avec le code qui l'entoure.

> Le terme _cargo cult programmer_ peut s'appliquer lorsqu'un programmeur informatique inexpérimenté avec le problème en question [copie du code](https://en.wikipedia.org/wiki/Copy_and_paste_programming) d'un endroit à un autre avec peu ou pas de compréhension de son fonctionnement ou de sa nécessité dans sa nouvelle position.  
>   
> — [Cargo Cult Programming sur Wikipedia](https://en.wikipedia.org/wiki/Cargo_cult_programming)

Lorsque qu'un développeur **copie** un morceau de code qu'il ne comprend pas et l'utilise dans l'espoir de résoudre un problème, il pratique la programmation Cargo Cult. Cela augmente le risque d'effets secondaires non intentionnels.

Lorsque qu'un développeur **lit** un morceau de code qu'il ne comprend pas et le modifie toujours dans l'espoir de résoudre un problème, il pratique également la programmation Cargo Cult.

Le problème, dans ce cas, n'est pas que le développeur copie quelque chose. N'importe qui peut copier un extrait de code, le comprendre, en apprendre, l'utiliser et avoir encore un risque plus faible d'effets secondaires non intentionnels.

La programmation Cargo Cult, en revanche, représente une incompréhension fondamentale des étapes éprouvées pour apprendre du code des autres.

Voici la manière la plus efficace d'apprendre dans ce contexte :

1. Lisez un morceau de code.
2. Comprenez toutes les fonctionnalités du langage qui sont utilisées.
3. Comprenez toutes les fonctionnalités des bibliothèques/frameworks qui sont utilisées.
4. Apprenez les bases de ces bibliothèques/frameworks.
5. Comprenez ce que fait chaque ligne et le but de ces bibliothèques/frameworks dans le contexte de ce morceau de code.

Pour quelqu'un qui commence à travailler avec un nouveau langage, cela va être extrêmement difficile. La quantité d'informations qu'un humain doit retenir pour comprendre efficacement même un petit extrait de code est si énorme qu'elle pourrait être oubliée immédiatement. Ce que nous pouvons faire, c'est tirer parti de certains aspects de la manière dont le cerveau humain est habitué à apprendre les choses — consciemment ou inconsciemment — avec certaines techniques éprouvées.

L'une de ces techniques est la [pratique bloquée](https://psychologywod.com/2013/08/18/blocked-practice-vs-random-practice-shake-things-up-in-your-training-and-in-your-life/). C'est essentiellement là où vous apprenez en « effectuant une seule compétence encore et encore, la répétition étant la clé ».

Ce n'est pas la meilleure façon d'apprendre, cependant. [Il est prouvé](https://psychologywod.com/2013/08/18/blocked-practice-vs-random-practice-shake-things-up-in-your-training-and-in-your-life/) que, lorsque vous apprenez en entrelaçant différentes variations de la même compétence, vous apprendrez plus efficacement.

En ingénierie logicielle, nous pouvons tirer parti des pratiques d'apprentissage bloquées et entrelacées lorsque nous **tapons** le code dans différents contextes, au lieu de simplement le **copier et coller**.

Lorsque nous copions-collons des extraits, nous ne faisons que lire (si nous prenons même la peine de le faire). Et selon la relation du Cône de l'Expérience, nous pourrions n'apprendre qu'une petite portion des informations que nous consommons parce que c'est trop abstrait.

Contrastez cela avec un apprentissage meilleur en tapant réellement ce morceau de code. C'est une expérience plus directe et intentionnelle. Cela force votre cerveau à comprendre tous ces différents motifs et à apprendre plus efficacement.

> Taper du code au lieu de le copier-coller offre un meilleur retour sur investissement en apprentissage parce que nous pratiquons au lieu de simplement lire.

Donner des noms aux choses est considéré comme l'un des aspects les plus difficiles de la programmation. Lorsque nous copions du code sans le comprendre, nous risquons de casser quelque chose en écrasant des noms de variables, des noms de fonctions ou des classes.

Si nous comprenons d'abord le code, puis le tapons avec nos propres mots, nous pouvons renommer les choses de manière à convenir à notre application et nous assurer de ne pas avoir de [collisions de noms](https://en.wikipedia.org/wiki/Naming_collision), même si le résultat final est fonctionnellement le même que le morceau de code sur lequel nous nous basons.

En outre, si nous copions le code d'un endroit de notre base de code à un autre, il y a une chance que nous copiions des jetons qui sont inutiles ou que nous oubliions de changer des jetons qui auraient dû être changés.

Prenons l'exemple suivant de snippet HTML :

```html
<label for="name"></label>
<div class="input-wrapper">
  <input type="text" id="name">
</div>
```

Lorsque nous dupliquons ce code pour créer une nouvelle entrée, nous risquons d'oublier de changer l'attribut _for_ de l'élément _label_, ce qui brisera le comportement prévu pour la nouvelle entrée.

```html
<label for="name"></label>
<div class="input-wrapper">
  <input type="text" id="name">
</div>

<label for="name"></label>
<div class="input-wrapper">
  <input type="password" id="password">
</div>
```

Cet exemple est intéressant car c'est le genre de fonctionnalité qui est difficile à tester, même avec une régression visuelle. Cela dépend fortement des tests statiques — comme une revue de code — pour s'assurer que le code a été écrit dans le but prévu. (Dans ce cas, pour propager les événements de la souris à l'entrée pour le même _id_ de l'attribut _for_ du label.)

La même chose se produit avec les tests. Lorsque nous copions un test déjà réussi au lieu d'en créer un nouveau à partir de zéro, nous risquons de ne pas changer les jetons nécessaires qui, autrement, feraient échouer ce test.

Dans ce cas, cependant, nous pouvons prévenir cette erreur en utilisant le [Développement Piloté par les Tests](https://medium.com/@fagnerbrack/why-test-driven-development-4fb92d56487c) — un état d'esprit basé sur la création d'un test échouant d'abord, puis la modification du code de l'application pour le faire passer. Cet état d'esprit nous permet d'être confiants que nous sommes moins susceptibles de manquer quelque chose ou de [créer des faux positifs](https://medium.com/@fagnerbrack/mocking-can-lean-to-nondeterministic-tests-4ba8aef977a0).

Au lieu de copier du code sans le comprendre, apprenez du code des autres et pratiquez-vous dessus. Cela maximisera votre retour sur investissement en apprentissage.

Après tout, la ressource la plus précieuse d'un développeur est le cerveau — pas les doigts.
