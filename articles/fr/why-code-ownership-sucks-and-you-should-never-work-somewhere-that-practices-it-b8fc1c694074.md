La propriété du code est nulle.

Elle limite le code et freine votre croissance en tant que développeur.

Voyons ce qu'est la propriété du code et pourquoi elle détruit les individus et les organisations.

### Qu'est-ce que la propriété du code

La **propriété du code** (code ownership) est le cas où une personne dans une organisation possède une base de code.

Cela signifie qu'une seule personne est responsable de l'ensemble du système. L'individu est responsable d'apporter des modifications au code, de prendre les décisions concernant la conception et de gérer les bugs.

Disons que le système est une application web. Avec la propriété du code, une seule personne dans l'entreprise possède cette application web. S'il y a un problème avec l'application web, le propriétaire du code est la personne qui doit le résoudre.

### Pourquoi est-ce nul ?

La propriété du code est mauvaise pour les propriétaires de code car elle freine leur croissance.

De plus, la propriété du code cause des problèmes pour l'organisation et pour les propriétaires de code.

Si personne ne sait comment un système fonctionne, personne ne peut donner de revues de code efficaces. Pire encore, le code pourrait ne pas être revu du tout.

Le propriétaire du code est alors la seule personne qui regarde jamais la base de code. Cela empêche les propriétaires de code d'apprendre de leurs collègues, ce qui est la meilleure façon d'apprendre au travail.

En tant que propriétaire de code, vous pourriez prendre d'horribles décisions de conception sans même le savoir.

#### La propriété du code tue la motivation

Il est facile de perdre sa motivation quand on possède une base de code.

Les gens ne se soucient pas que vous ayez réduit la taille du bundle de 10 ko s'ils ne savaient même pas quelle était la taille du bundle auparavant.

Il n'y a personne pour s'enthousiasmer avec vous lorsque vous passez à un nouveau Framework, ajoutez [webpack](https://webpack.js.org/) au pipeline, ou améliorez les performances du serveur en ajoutant un cache.

C'est démotivant.

#### La propriété du code vous rend paresseux

Quand vous possédez votre code, vous pouvez devenir paresseux.

Personne ne vous surveille. Les gens ne savent pas que le bug ne vous a pris que cinq minutes à corriger, et que vous avez passé le reste de la journée sur [reddit](https://www.reddit.com/).

Vous pouvez vous laisser porter, en faisant le strict minimum. C'est un gâchis.

J'ai vu des trentenaires qui occupent le même emploi depuis dix ans, travaillant sur la même base de code héritée (legacy). Et cela se voit. Ils sont en retard sur les dernières technologies, ils ne sont pas passionnés par leur travail et ils passent des heures chaque jour à naviguer sur les réseaux sociaux.

Ne soyez pas comme eux.

### La propriété du code produit une dépendance à une personne clé

Non seulement la propriété du code est mauvaise pour l'individu, mais elle est aussi mauvaise pour l'organisation.

Si une seule personne est responsable d'une base de code, que se passe-t-il lorsque le propriétaire du code part en vacances — ou pire — quitte l'entreprise ? Personne dans l'organisation ne saura comment le code fonctionne.

Ce problème est connu sous le nom de **dépendance à une personne clé** (key man dependency). La propriété extrême du code produit une dépendance à une personne clé.

#### La propriété du code cause du stress

Bien qu'elle puisse vous rendre complaisant, la propriété du code cause aussi du stress.

S'il y a un problème avec l'application, c'est de votre faute. Vous serez responsable et vous n'aurez personne pour vous aider à corriger le bug.

### Quelle est la solution ?

La solution est une culture de **code partagé**.

Le code partagé est le cas où tout le monde dans l'équipe travaille sur toutes les parties de la base de code.

Lorsque l'équipe possède le code, tout le monde peut aider à prendre les décisions de conception. Tout le monde peut participer à la discussion sur la conception du système, aider à échanger des idées et partager la responsabilité de ces décisions. Écrire du code devient une expérience magique.

Avec le code partagé, il n'y a pas de dépendance à une personne clé. N'importe quel membre de l'équipe peut partir en vacances et l'équipe continuera à travailler normalement.

Avec le code partagé, vous apprendrez de vos pairs. Quelqu'un ayant connaissance du système peut fournir des commentaires de qualité. Le code sera examiné par de nombreux yeux, et il s'épanouira grâce à cela.

Lorsque plusieurs personnes travaillent sur la même base de code, elles aident à sculpter un code vers la perfection.

### Tirer le meilleur parti de votre temps au travail

Vous n'avez pas besoin de travailler dans une organisation qui freine votre croissance. Vous devriez tirer le meilleur parti de votre temps au travail.

Les développeurs ont le dessus sur le marché du travail. Nous pouvons choisir où nous travaillons. Si nous n'apprenons pas à notre poste actuel, nous pouvons changer d'emploi.

Si vous travaillez dans un endroit avec une propriété du code, trouvez un nouvel emploi.

Ne faites pas l'erreur de travailler dans un environnement sous-optimal. Cela vous nuira à long terme.

Vous ne réalisez peut-être pas à quel point cela affecte votre croissance, mais un endroit qui pratique la propriété du code vous empêchera d'atteindre votre plein potentiel.

Apprendre d'autres développeurs est un grand avantage du travail en équipe, alors trouvez un nouvel emploi dès maintenant.
