Je trouve incroyable le nombre de significations différentes que nous pouvons obtenir avec moins de 30 caractères. Je parle bien sûr de l'alphabet avec une ponctuation bien placée. D'une histoire d'amour à un programme informatique, l'écriture nous a permis de créer des mondes extraordinairement différents. Et le langage, en général, fournit un cadre dans lequel nous pouvons accrocher un flux d'idées.

La plupart des langages de programmation semblent avoir des normes très strictes — certains termes doivent être utilisés à certains endroits. Mais il y a un domaine où il y a une énorme liberté : la façon dont nous nommons ces termes.

Prenons ce simple programme qui crée une phrase à partir d'un tableau de mots, ajoute une marque de ponctuation et l'enregistre dans la console :

![Image](https://cdn-media-1.freecodecamp.org/images/dRQKspy4V21MPbO8LW1bFXtIOwTMvnNeRp5i align="left")

Rien de trop spécial, n'est-ce pas ? Mais ce que vous n'avez peut-être pas considéré, c'est le nombre de *termes* dont vous étiez responsable pour nommer.

Il y a 23 mots (sans inclure les valeurs codées en dur) dans le programme ci-dessus. Nous avons contrôlé les noms de 14 de ces mots. Cela représente plus de 60 % de ce qui a été tapé, et c'était notre responsabilité de les nommer !

![Image](https://cdn-media-1.freecodecamp.org/images/rQnG17-jJTsPsKtn0NCIPVD60qKRR8nhHNMX align="left")

*Vous êtes responsable de la majorité des noms dans votre programme*

Que vous construisiez une application d'entreprise ou un simple *Bonjour, Monde*, vous voulez que votre programme se lise comme un best-seller du Times, et non comme un cahier de Mad Libs.

Ceci n'est *pas* un manifeste sur la façon de structurer une application JavaScript entière, mais plutôt un chapitre sur la façon de choisir des noms pour ces choses dans lesquelles vous avez la liberté de le faire. J'ai inclus les diverses références en bas si vous souhaitez en savoir plus. Une dernière chose, le point clé de toutes nos conventions et normes est le suivant :

> *La plupart de ces conventions ne sont pas pour vous aujourd'hui, mais plutôt pour vous et les personnes qui liront votre code demain.*

![Image](https://cdn-media-1.freecodecamp.org/images/-Se0xzoDIzsROxX4YKSWtQuD6Z7Mq4UbT-ub align="left")

#### Références et apprentissage continu

1. ["Clean Code" par Robert C. Martin](https://www.amazon.com/gp/product/0132350882/ref=as_li_tl?ie=UTF8&camp=1789&creative=9325&creativeASIN=0132350882&linkCode=as2&tag=brandonwoz-20&linkId=8af093cb2b8d9a87993f285341ff015a) — *Une excellente lecture pour tous les langages. Il va au-delà des conventions de nommage et propose la structure entière de votre programme. Les exemples sont en Java, mais les principes s'appliquent à JavaScript.*

2. ["Clean Code JavaScript" par Ryan Mcdermott](https://github.com/ryanmcdermott/clean-code-javascript) — *Le livre ci-dessus, mais remanié pour JavaScript. Il est disponible en ligne et est gratuit.*

3. [Guide de style JavaScript d'Airbnb](https://github.com/airbnb/javascript) — *Possiblement le guide de style le plus complet pour JavaScript. Il contient non seulement le quoi, mais aussi le pourquoi. (Si vous êtes sur un petit appareil, vous devrez peut-être cliquer sur "view all readme" pour voir le document entier).*

4. [Guide de style JavaScript W3](https://www.w3schools.com/js/js_conventions.asp) — Un guide _court et concis_.

5. [Guide de style ES6 de Google](https://google.github.io/styleguide/jsguide.html) — *Le guide de style de Google pour JavaScript.*
