Par Bertil Muth

Dans mon travail avec les équipes agiles et en tant que formateur, j'entends souvent des affirmations sur les user stories qui posent problème en pratique. Il est temps de se débarrasser des mythes les plus courants.

## Mythe #1 : Dans Scrum, vous documentez les exigences avec des user stories.

Le Guide Scrum laisse ouvert la manière dont les éléments du backlog sont documentés. J'aime les user stories parce qu'elles déplacent la perspective du système vers l'utilisateur. Mais elles ne sont pas obligatoires. Et elles devraient être utilisées davantage pour la conversation que pour la documentation.

## Mythe #2 : Si vous écrivez des user stories, vous devez utiliser un certain modèle.

Vous pensez probablement à : En tant que _utilisateur_, je veux _fonctionnalité_ afin que _besoin satisfait_.

Le concept de user story n'avait rien à voir avec ce modèle au début. Plus tard, le modèle est devenu populaire. Il est possible de documenter les user stories de manière informelle. Ne vous attardez pas sur le modèle.

## Mythe #3 : Les user stories sont estimées avec des Story Points.

En tant que Scrum Master, j'ai vu des équipes devenir folles avec des discussions sur les story points, les estimations et la vélocité.

Il existe une alternative simple. L'équipe se met d'accord sur des petites stories. Donc 1 ou 2 jours pour les implémenter. Ensuite, après le sprint, l'équipe compte les stories qu'elle a terminées. Voici la vélocité.
